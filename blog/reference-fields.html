Adding reference fields to forms without a data source.

Every D365 F&O consultant hits this eventually: a form where someone needs a lookup that isn't backed by a data source, and the form designer flatly refuses. Here's the cleanest pattern I've landed on after a decade of running into it.

Contents

  2. The pattern, in one picture
  3. The code

#Why the form won't let you

Reference groups — the lovely control that gives you a name-plus-description lookup for free — need a data source field of type EDT backed by a surrogate foreign key relation. If the field isn't on a data source (think: an unbound parameter dialog, a SysOperation contract, or a custom tmp form you've stitched together in X++), the designer will refuse to let you drop the control.

The instinct is to add a data source just to satisfy the designer. Don't. You end up fighting cursor behaviour, validation, and inserts for something that was never meant to persist.

i If the field will eventually persist — e.g. you're capturing it to run a later operation — a proper data source is still the right answer. This post is for the transient, dialog-only case.

#The pattern, in one picture

Build the reference group manually out of two plain controls — a StringEdit for the key and a StringEdit for the description — then override lookup() and validate() on the first control. That's it. The designer never has to know. 

X++// Dialog class — SysOperation contract / RunBase descendant
public class MyParmDialog extends RunBaseBatch
{
    CustAccount custAccount;
    DialogField  dialogCustAccount;

    public Object dialog()
    {
        DialogRunbase dlg = super();

        dialogCustAccount = dlg.addFieldValue(
            extendedTypeStr(CustAccount),
            custAccount,
            "Customer",
            "Pick a customer — lookup is wired below.");

        return dlg;
    }
}

Then, in the form wrapping that dialog (or with a FormBuildControl programmatically), attach a lookup handler: 

X++public void lookup(FormStringControl _control)
{
    Query                   q = new Query();
    QueryBuildDataSource   qbds;
    SysTableLookup         lookup;

    qbds = q.addDataSource(tableNum(CustTable));
    qbds.addSortField(fieldNum(CustTable, AccountNum));

    lookup = SysTableLookup::newParameters(tableNum(CustTable), _control);
    lookup.addLookupField(fieldNum(CustTable, AccountNum));
    lookup.addLookupField(fieldNum(CustTable, Name));
    lookup.parmQuery(q);
    lookup.performFormLookup();
}

#The code

The full example — dialog contract, lookup override, and the description-fill in modified() — lives in a gist linked at the bottom. The two things that are easy to get wrong:

  • Use SysTableLookup::newParameters, not the older SysTableLookup::new. The former wires the selected-value callback for you.
  • If you need the "description" half to update live, override modified() on the key control and fillLookup()-style read the related record yourself.

Reference groups are a convenience. They're not a law of nature. The day the convenience stops fitting your case, drop down a level and build the two controls yourself — you already know how.

#Gotchas in production

  1. Security. Manual lookups bypass the datasource's query security. If your lookup hits a sensitive table, wrap it with the same QueryBuildRanges the data source would have applied.
  2. Validation. Without a data source, the framework won't call your validateWrite. Do the existence check explicitly in the control's validate() method.
  3. Label leaks. Labels on bare StringEdits default to the EDT label — which is fine — but if you've extended the EDT, the change may not propagate. Pin the label on the control directly.
! Never cache the selected record in a form-level static. I've watched one of those outlive three users across a Terminal Services session and poison a batch. Use parmMap or a per-instance field.

#Wrap-up

You don't need a data source to get a reference-group-shaped experience. Two string controls, a SysTableLookup, and a careful modified() are enough — and they keep the dialog honest about the fact that nothing here persists until the operation runs.
